<h1>Introducing MIMERCode: A Programming Language Built for Humans and AI</h1>
<!-- /wp:heading -->

<!-- wp:paragraph -->
<p><em>What happens when you design a programming language from scratch, knowing that half the code written in it will be generated by an AI?</em></p>
<!-- /wp:paragraph -->

<!-- wp:paragraph -->
<p>That question drove the creation of <strong>MIMERCode</strong> &mdash; a new programming language that is readable by business analysts, writable by ChatGPT and Claude, and secure enough that you can execute AI-generated code without losing sleep.</p>
<!-- /wp:paragraph -->

<!-- wp:paragraph -->
<p>MIMERCode ships as two executables: <strong>MIMERCodeRunner</strong> for console programs and <strong>MIMERCodeRunnerGUI</strong> for graphical applications. Write a <code>.mc</code> file, double-click, and it runs. No compiler toolchain. No package manager. No dependency hell.</p>
<!-- /wp:paragraph -->

<!-- wp:heading -->
<h2>Why Another Language?</h2>
<!-- /wp:heading -->

<!-- wp:paragraph -->
<p>We already have Python, JavaScript, and a hundred others. So why MIMERCode?</p>
<!-- /wp:paragraph -->

<!-- wp:paragraph -->
<p>Because the world changed. Large Language Models now generate code every day &mdash; inside chatbots, inside automation pipelines, inside tools that non-programmers use to build things. And the languages those LLMs target were designed decades ago, for humans typing at keyboards. They are full of ambiguity, implicit behavior, and silent failure modes that LLMs stumble over constantly.</p>
<!-- /wp:paragraph -->

<!-- wp:paragraph -->
<p>MIMERCode was designed with three goals:</p>
<!-- /wp:paragraph -->

<!-- wp:list {"ordered":true} -->
<ol>
<li><strong>Readable by everyone.</strong> Plain English keywords. No semicolons. No invisible whitespace rules. Every block closes with <code>end</code>. A business analyst can read MIMERCode and understand what it does.</li>
<li><strong>Writable by AI.</strong> The syntax deliberately matches what LLMs already know from Python and JavaScript. F-strings, slicing, dictionaries, lambdas &mdash; they all work the way an LLM expects. Case-insensitive keywords mean <code>True</code>, <code>true</code>, and <code>TRUE</code> all work.</li>
<li><strong>Secure by design.</strong> Contracts validate data at boundaries. Channels sandbox all I/O. Sensitive fields are sealed automatically. There is no way for code &mdash; human or AI-generated &mdash; to silently access files, networks, or databases without explicit permission.</li>
</ol>
<!-- /wp:list -->

<!-- wp:heading -->
<h2>What Does It Look Like?</h2>
<!-- /wp:heading -->

<!-- wp:paragraph -->
<p>Here is a complete MIMERCode program:</p>
<!-- /wp:paragraph -->

<!-- wp:code -->
<pre><code>program HelloWorld
  local Name := "World"
  WriteLn(f"Hello, {Name}!")

  for I := 1 to 5 do
    WriteLn(f"Count: {I}")
  end
end</code></pre>
<!-- /wp:code -->

<!-- wp:paragraph -->
<p>Save it as <code>hello.mc</code>, run <code>MIMERCodeRunner hello.mc</code>, and you get:</p>
<!-- /wp:paragraph -->

<!-- wp:code -->
<pre><code>Hello, World!
Count: 1
Count: 2
Count: 3
Count: 4
Count: 5</code></pre>
<!-- /wp:code -->

<!-- wp:paragraph -->
<p>No imports. No boilerplate. No configuration files. If you have ever written a line of code in any language, you can read this.</p>
<!-- /wp:paragraph -->

<!-- wp:heading -->
<h2>Contracts: Data You Can Trust</h2>
<!-- /wp:heading -->

<!-- wp:paragraph -->
<p>Every real program receives data from the outside world &mdash; from users, APIs, databases. And that data is frequently wrong. MIMERCode solves this with <strong>contracts</strong>: you declare the exact shape of valid data in one place, and the runtime enforces it automatically.</p>
<!-- /wp:paragraph -->

<!-- wp:code -->
<pre><code>contract Customer
  Name:   required String minlen(1) maxlen(200),
  Email:  required String,
  Age:    optional Integer range(0, 150),
  Active: default true Boolean
end</code></pre>
<!-- /wp:code -->

<!-- wp:paragraph -->
<p>If invalid data arrives, it is rejected <em>before your code ever sees it</em>. No manual validation. No scattered if-checks. The contract is simultaneously the definition, the validation, and the documentation.</p>
<!-- /wp:paragraph -->

<!-- wp:paragraph -->
<p>Sensitive fields get even better treatment:</p>
<!-- /wp:paragraph -->

<!-- wp:code -->
<pre><code>contract PaymentRequest
  CardNumber: required String sensitive,
  CVV:        required String sensitive,
  Amount:     required Double gte(0)
end</code></pre>
<!-- /wp:code -->

<!-- wp:paragraph -->
<p>Fields marked <code>sensitive</code> are <strong>sealed at creation</strong>. Every accessor returns masked data (<code>****</code>). Logging cannot accidentally leak card numbers. Error messages cannot expose secrets. The only way to access the real value is an explicit <code>unseal()</code> call, creating a clear audit trail.</p>
<!-- /wp:paragraph -->

<!-- wp:heading -->
<h2>Pipelines: Composable Workflows</h2>
<!-- /wp:heading -->

<!-- wp:paragraph -->
<p>Real-world programs are multi-step workflows: validate, enrich, transform, render, deliver. MIMERCode pipelines make this explicit:</p>
<!-- /wp:paragraph -->

<!-- wp:code -->
<pre><code>pipeline InvoiceProcess
  input as OrderData
  output as DeliveryPackage

  Validated := ValidateOrder(Input)

  Customer, Pdf := fork Validated
    =&gt; EnrichCustomer()
    =&gt; RenderOrder() |&gt; HtmlToPdf()
  end

  return Assemble(Customer, Pdf)
end</code></pre>
<!-- /wp:code -->

<!-- wp:paragraph -->
<p>Each step has a name and a contract. The <code>fork</code> keyword runs branches in parallel. Every step can declare its own error policy: abort, skip, retry, or fallback. The runtime logs every step with timing and trace IDs automatically. You can even load pipeline definitions from JSON configuration &mdash; change the workflow without changing code.</p>
<!-- /wp:paragraph -->

<!-- wp:heading -->
<h2>Channels: One Pattern for All I/O</h2>
<!-- /wp:heading -->

<!-- wp:paragraph -->
<p>Files, HTTP, databases, authentication &mdash; MIMERCode uses the same two-verb pattern for everything:</p>
<!-- /wp:paragraph -->

<!-- wp:code -->
<pre><code>// Read a file
local F := open("file://data/config.json")
local Content := F.Read()
close(F)

// Call an API
local Api := open("https://api.example.com/v1")
local Users := Api.Get("/users")
close(Api)

// Serve HTTP
local Web := serve("http://0.0.0.0:8080")
on Web.Get("/") as Req: HttpRequest
  return "&lt;html&gt;&lt;h1&gt;Hello!&lt;/h1&gt;&lt;/html&gt;"
end</code></pre>
<!-- /wp:code -->

<!-- wp:paragraph -->
<p>All paths are sandboxed. Path traversal is rejected. There is no <code>eval()</code>, no <code>exec()</code>, no <code>import os</code>. If an LLM generates MIMERCode, it physically cannot access resources you have not explicitly permitted through a channel.</p>
<!-- /wp:paragraph -->

<!-- wp:heading -->
<h2>A Complete GUI Framework</h2>
<!-- /wp:heading -->

<!-- wp:paragraph -->
<p>MIMERCode includes a full declarative GUI framework, rendered by its own canvas engine &mdash; every pixel is painted by MIMERCode, with no platform widget dependencies.</p>
<!-- /wp:paragraph -->

<!-- wp:code -->
<pre><code>view Counter
  state
    Count := 0
  end
  func Increment()
    Count := Count + 1
  end
  render
    vbox padding: 16, gap: 8
      label text: f"Count: {Count}", font_size: 24 end
      button text: "Increment", on_click: Increment end
    end
  end
end

local App := serve("gui://Counter",
  title: "My Counter", width: 300, height: 200
)</code></pre>
<!-- /wp:code -->

<!-- wp:paragraph -->
<p>Over 30 built-in widgets are available: text inputs, textareas with line numbers, numeric editors with hex/binary support, checkboxes, radio buttons, sliders, dropdown selects, tab containers, splitters, menubars with keyboard shortcuts, tree views, status bars, and more.</p>
<!-- /wp:paragraph -->

<!-- wp:paragraph -->
<p>The <strong>table</strong> widget is a full data grid with sorting, column filtering, cell editing, grouping, custom cell templates, and keyboard navigation:</p>
<!-- /wp:paragraph -->

<!-- wp:paragraph -->
<p>The <strong>chart</strong> widget renders line, bar, area, pie, donut, and scatter charts with tooltips, legends, zoom, and click events. Multi-series combo charts (bar + line overlay) are supported with explicit series blocks.</p>
<!-- /wp:paragraph -->

<!-- wp:paragraph -->
<p><strong>Choropleth map charts</strong> display geographic data with color-coded regions, hover effects, and drill-down navigation. Click a country to see its states. Click a state to see its counties. The built-in map downloader tool converts Natural Earth GeoJSON data to MIMERCode's lightweight <code>.mcmap</code> format.</p>
<!-- /wp:paragraph -->

<h2>Animations That Just Work</h2>
<!-- /wp:heading -->

<!-- wp:paragraph -->
<p>Add a single <code>transition</code> property to any widget and style changes animate smoothly:</p>
<!-- /wp:paragraph -->

<!-- wp:code -->
<pre><code>button text: "Hover me",
  background: if IsHovered then "primary" else "surface" end,
  transition: "button_hover",
  on_mouseenter: func() IsHovered := true end,
  on_mouseleave: func() IsHovered := false end
end</code></pre>
<!-- /wp:code -->

<!-- wp:paragraph -->
<p>Thirteen named transition presets are built in &mdash; from subtle <code>"fade"</code> and <code>"focus"</code> to playful <code>"bounce"</code> and <code>"spring"</code> with elastic overshoot. Twelve easing functions are available, including cubic, back, bounce, and elastic variants. All animations run at 60fps.</p>
<!-- /wp:paragraph -->

<!-- wp:paragraph -->
<p>Full SVG rendering with SMIL animations is also supported &mdash; including <code>&lt;animate&gt;</code>, <code>&lt;animateTransform&gt;</code>, and <code>&lt;animateMotion&gt;</code> for path-following animations. Build loading spinners, animated icons, and data visualizations directly in SVG.</p>
<!-- /wp:paragraph -->

<h2>Math, Statistics, Geography, and Data Formats</h2>
<!-- /wp:heading -->

<!-- wp:paragraph -->
<p>The standard library is comprehensive:</p>
<!-- /wp:paragraph -->

<!-- wp:list -->
<ul>
<li><strong>55+ math functions</strong> &mdash; trigonometry, logarithms, combinatorics, random distributions (normal, uniform), bitwise operations, and constants (<code>pi</code>, <code>e</code>, <code>tau</code>, <code>inf</code>, <code>nan</code>)</li>

<li><strong>Data formats</strong> &mdash; JSON parse/stringify and CSV parse/stringify with full RFC compliance (quoted fields, escaped quotes, custom delimiters)</li>
<li><strong>Templates</strong> &mdash; Mustache-based rendering with automatic sensitive field masking</li>
</ul>

<!-- wp:heading -->
<h2>For LLM System Builders</h2>
<!-- /wp:heading -->

<!-- wp:paragraph -->
<p>If you are building a system where an LLM generates code at runtime, MIMERCode was designed specifically for you. The combination of contracts, channels, and pipelines means you can execute LLM-generated code with confidence:</p>
<!-- /wp:paragraph -->

<!-- wp:list -->
<ul>
<li><strong>Invalid data is rejected at entry</strong> &mdash; contracts validate before your code runs</li>
<li><strong>Secrets never leak</strong> &mdash; sensitive fields are sealed, even in error messages</li>
<li><strong>I/O is sandboxed</strong> &mdash; channels prevent unauthorized file, network, or database access</li>
<li><strong>Every step has error handling</strong> &mdash; pipelines provide per-step abort/skip/retry/fallback</li>
<li><strong>Execution is fully traceable</strong> &mdash; built-in logging with trace IDs and timing</li>
</ul>
<!-- /wp:list -->

<!-- wp:paragraph -->
<p>An LLM that generates MIMERCode cannot <code>import os; os.system("rm -rf /")</code>. It cannot read <code>/etc/passwd</code>. It cannot send your data to an unauthorized endpoint. The language itself makes these things impossible.</p>
<!-- /wp:paragraph -->

<!-- wp:heading -->
<h2>For Delphi Developers</h2>
<!-- /wp:heading -->

<!-- wp:paragraph -->
<p>MIMERCode will feel immediately familiar. It inherits Delphi's class/record/interface model, constructor/destructor lifecycle, virtual/override dispatch, properties, and try/except/finally. What changed is the boilerplate: no <code>begin</code>/<code>end</code> pairs (just <code>end</code>), no semicolons, <code>func</code> instead of <code>function</code>/<code>procedure</code>, <code>return</code> instead of <code>Result :=</code>, and <code>local</code> variables declared anywhere instead of a <code>var</code> block at the top.</p>
<!-- /wp:paragraph -->

<!-- wp:paragraph -->
<p>The runtime is built in Delphi, rendered via GDI+, and runs natively on Windows.</p>
<!-- /wp:paragraph -->
